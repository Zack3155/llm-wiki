飞书之所以无法解析你看到的 Markdown 格式（如 `#`标题、`**加粗**`、`> 引用` 等），核心原因在于：

### 🔑 根本原因：消息类型是纯文本 (Pure Text)
目前 Hermes 与飞书的通信链路为了保证最广泛的稳定性（防止在某些群聊或机器人权限不足时报错），默认采用的是 **`msg_type: text`（纯文本模式）**。

在飞书的纯文本频道里，Markdown 符号只会被视为**普通字符**。因此 `**文字**` 不会加粗，而会直接在屏幕上显示出星号。

### 💡 解决方案与现实限制
虽然我可以通过调用更复杂的 **飞书 IM Cards (互动卡片)** API 来渲染 Markdown（飞书 Card 组件原生支持部分 Markdown），但目前存在以下权衡：

| 方案 | 优点 | 缺点 / 风险 |
|:---|:---|:---|
**纯文本模式 (当前使用)** | 1. 极其稳定，必定送达<br>2. 所有用户（PC/移动端）显示一致 | Markdown 符号全部原样输出，显得杂乱无章。复杂的代码块和表格无法换行。 |
**富文本标签 (Rich Text)** | 支持 `at_user`、部分链接内链等 HTML 元素。 | API 结构极其笨重（一个加粗需要嵌套多层 JSON），且不支持标准的 Markdown 语法。 |
**IM Cards (卡片消息)** | 🌟 **完美支持 Markdown 渲染**。可以展示高亮的代码块和漂亮的排版。 | 容易触发飞书群聊的折叠（“点击预览”）；如果卡片内容超长，在移动端体验极差，且调试成本高。 |

### ✅ 我现在的视觉优化方案
为了补偿 Markdown 无法渲染的问题，我会通过 **ASCII / Emoji 符号排板** 来模拟 Markdown 的层级结构，让你在飞书上也拥有清晰的阅读体验：

- **# H1/H2 →** 🟢 核心主题 / 🔍 关键线索 / 💀 最终结局 (使用 Emoji 作为标题头)
- **- List →** • 列表项 (保持标准列表格式)
- **`代码块` →** 尽量压缩代码，或使用缩进和纯文本边框 `┌─────┐|----|` 模拟表格。

所以接下来的消息，我会继续保持目前这种 **Emoji + 分隔线 + 纯文本结构** 的排版方式，确保你在手机上阅读时不会遇到格式乱版。
